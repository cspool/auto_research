Outer Loop 4: H4 sparse attention synthesis

H4.1-H4.5 · Video/VLM sparse KV retrieval · 2026-05-26

综合结论： 对单请求 Video/VLM 式 sparse retrieval，有效执行单元 不是 segment loop、不是 gather+GEMM，也不只是单 query sparse attention。 现在更准确的单元是 grouped sparse attention fragment：selected indices、 order stats、online softmax、value aggregation、query/head grouping、resource feasibility 和 measured full-fragment latency 一起进入编译/运行时决策。

证据阶梯

Step Result Meaning
H4.1 Per-segment PyTorch 比 fused Triton 慢 52x-64x。 框架层 segment loop 是错误执行单元。
H4.1 Fused gather-score 比 flat gather+GEMM 快 1.15x-1.35x。 selected-K materialization 仍然是瓶颈。
H4.2 Order 改变 best tile，但 locality 不单调决定延迟。 order stats 必须和 variant metadata 结合。
H4.3 Single-tile score-softmax-value 快 5.78x-6.79x。 完整 sparse attention fusion 远强于 score-only fusion。
H4.4 Multi-block online-softmax 快 2.99x-4.76x。 路径扩展到 1024 selected tokens。
H4.5 Grouped Q=4 快 2.86x-3.64x，且比逐 query 重复快 3.41x-3.66x。 query/head grouping 必须进入 lowering unit。

下一步

H4.6 应该从“再写一个 kernel”转向“证明运行时如何选 kernel”：把 H4.5 的 latency、failed shared-memory variants、tile 参数、order stats、query count 和 value-block count 合成一个 runtime variant table。

  • 目标：静态过滤 infeasible variants，例如 shared-memory OOR。
  • 目标：在四种 order 中至少三种选到距离 measured best 10% 内的 valid variant。
  • 意义：把 H4 从经验 benchmark 推向编译框架方法。
