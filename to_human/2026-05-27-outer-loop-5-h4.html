Outer Loop 5: H4 runtime selector synthesis

H4.1-H4.9 · grouped sparse attention · 2026-05-27

综合结论： H4 已经从“写一个更快 sparse attention kernel” 变成了完整的编译/运行时方法：把 selected-token retrieval 降到 grouped sparse attention fragment，再用静态资源过滤、order stats、query pressure 和 measured full-fragment latency 选择 tile/warp variant。
8 / 8 H4.8 在 held-out query-count 上 exact best。
8 / 8 H4.9 在 512/2048 selected tokens 上 15% 内 near-best。
16 / 16 H4.7+H4.9 observed OOR rows 被静态 feasibility 过滤。

证据阶梯

Step Question Result Meaning
H4.1-H4.3 能否把 Video/VLM sparse retrieval 降到 fused sparse attention? 从 52x-64x loop gap 到 5.78x-6.79x score-softmax-value speedup。 segment loop 和 gather materialization 都不是合适执行单元。
H4.4-H4.5 能否扩展到 multi-block 和 grouped query? 1024 selected tokens 仍快 2.99x-4.76x；Q=4 grouped 快 2.86x-3.64x。 online softmax 和 query/head grouping 必须进入 lowering unit。
H4.6-H4.8 runtime table 能否选择 variant? 静态过滤 OOR；query-aware selector 修复 Q=8 clustered miss。 selector 需要 order stats + query pressure + measured latency。
H4.9 selected-token count 是否是新的 failure axis? 512/2048 selected tokens 上 8/8 near-best，mean regret 4.66%。 selected-token count 是 exact tuning feature，暂不是 near-best failure。

当前方法

  • selected-token set、segment boundary 和 order stats 作为 metadata。
  • K/V movement、score、online softmax、value aggregation 合成 sparse attention fragment。
  • 共享 K/V 的多 query/head 被 grouped 到同一 lowering unit。
  • shared-memory feasibility 先过滤 invalid variant，再用 runtime table 选择 measured near-best。

下一步

继续深化一次：H4.10 做 value-dimension held-out shapes。推荐固定 selected_tokens=1024、Q=4、四种 order，测试 V=64 和 V=256，并保留 BLOCK_V=64/128 的 variant 集合。

  • 如果 H4.8/H4.9 selector 在 V=256 上 miss，说明 value pressure 是下一类 table key。
  • 如果仍然 near-best，H4 selector story 足够进入 learned/table selector 或硬件/NPU mapping。
  • 硬件解释重点转向 selected-token movement、online reduction、value partial-buffer traffic。
