单加速器单请求推理：多算子/微算子并发

Local-notes bootstrap report · 2026-05-25

核心结论： 单请求场景下，单纯把框架算子丢进多个 stream 通常不够。 更可靠的路线是“语义拆分 + 微算子/多版本 kernel + 硬件资源感知调度/融合”。

方法谱系

层次代表系统对本课题的启发
运行时共置与空间划分 μShare, Bullet, EEVEE 通过 MPS、SM mask、block-size shaping、launch timing 控制并发。
编译生成微算子 Infera, STOF, MetaAttention 把 operator 切成 tile/micro-operator，并生成可调度的多版本 kernel。
核内融合与 inter-core 通信 FlashFuser, PAT, JanusQuant 把并发移入 fused kernel，减少 launch 和 global memory round trip。
模型语义拆分 Difflow, MixFusion, FineMoE, LWD 利用 expert、patch、dGraph、denoising step、wavelet saliency 等语义单位。
硬件/编译协同 Focus, V-Rex, RPU, ASM-SpMM 对不规则 token/KV/稀疏访问，硬件流水和后端 intrinsic 往往比 GPU stream 更有效。

初始假设

  1. H1: 单请求中，multi-stream 只有在资源互补或空间受控时才有明显收益。
  2. H2: 微算子切分 + 多版本 kernel 会优于粗粒度通用融合。
  3. H3: MoE、Diffusion/DiT、多模态、Video 的语义拆分能暴露通用编译器看不到的并发。
  4. H4: 视频/VLM 的不规则检索和 token 选择更可能需要硬件感知或硬件辅助。
